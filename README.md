## Slots

Для меня в данной задаче основным приоритетом был выбор стратегии кеширования. Поскольку в задаче участвуют 
временные слоты, а это критичные данные с точки зрения использования. Например, пользователь выбирает слот на 
определенное время, время ему показано, но захолдировать он не может по факту. В связи с этим, естественно 
использовать только Mutex. Может, конечно, я сильно воспринял критичность данных. Например, если бы это был какой-то 
контентный проект, то можно было использовать стратегию Stale While Revalidate, когда фоном идет обновление.

В целом использую редис для кеша, но и memcached вполне для такой задачи подойдет. Но, конечно, зависит от общего 
объема и нужно ли что-то все же сбрасывать на диск. Но в данном примере пусть будет редис.

## Как запустить

- git clone git@github.com:backinvertedzero/slots-laravel-demo.git
- cd slots-laravel-demo
- docker compose -p slots -f docker/docker-compose-local.yml up -d

В целом привык использовать докер, чтобы запускать подобные проекты. Не иметь на ноуте какую-то одну базу, например. 
И тому подобные вещи. А контейнеры перестали быть нужны - прибил их и ресурсы освободились.
Но помимо этого здесь я не стал учитывать персистентность хранилища. То есть если контейнер mysql прибить, то 
таблицы не будут сохранены.

## Подготовка данных
Чтобы запустить миграции и накатить сидер, для этого нужно зайти в контейнер приложения
- docker exec -it slots_app /bin/bash


Уже в самом контейнере выполнить
- php artisan migrate
- php artisan db seed

Можно, конечно, придумать автоматизацию, но да ладно.
Приложение пишет логи. Соответственно логи тоже можно смотреть в контейнере приложения
